# Frontend Design Language

## 1. 문서 목적

- 이 문서는 프론트엔드 시각 언어의 최상위 원칙을 정의한다.
- 구현 상세는 아래 하위 문서에서 관리한다.
- 토큰 사전: `docs/tokens-spec.md`
- 컴포넌트 계약: `docs/component-contract.md`
- 전환 절차: `docs/migration-playbook.md`

## 2. 범위

- 대상: `frontend`
- 포함: Report UI, Recommender UI 공통 디자인 언어
- 제외: 백엔드 분석 로직, 데이터 스키마

## 3. 브랜드 방향

### 3.1 핵심 톤

- 시각적 기조는 Steam과 유사한 다크 네이비 톤을 유지한다.
- 강조색은 블루 계열을 중심으로 사용한다.
- 전체 분위기는 "차분함 + 정보 신뢰성 + 판단 보조"를 목표로 한다.

### 3.2 금지 사항

- 밝은 배경 중심의 라이트 톤으로 기조를 전환하지 않는다.
- 포인트 색상을 보라/네온 계열로 임의 교체하지 않는다.
- 장식용 그래디언트/그림자를 과도하게 사용하지 않는다.

## 4. 디자인 원칙

- 정보 전달 우선: 장식보다 정보 위계를 먼저 설계한다.
- 일관성 우선: 토큰을 통해 화면 간 변동을 줄인다.
- 의미 우선: 상태 색상은 감정 표현이 아니라 사용자 판단 보조를 위한 수단으로 사용한다.
- 접근성 기본값: 대비, 포커스, 키보드 탐색을 기본 요구사항으로 둔다.

## 5. 토큰 운영 원칙

- 모든 시각값은 가능한 한 `src/styles/tokens.css`를 통해 사용한다.
- 컴포넌트 CSS/JSX에서 raw color 값(`hex`, `rgb`) 직접 사용을 원칙적으로 금지한다.
- 토큰은 계층별 역할을 지킨다.
- `ref`: 브랜드 팔레트 원본값
- `sys`: 의미 기반 시스템 역할
- `comp`: 컴포넌트 전용 역할
- `semantic`: 도메인 상태 표현(`buy-*` 등)
- 기존 변수(`--bg`, `--card`, `--brand`)는 호환 레이어로 유지하되, 신규 개발은 `sys`/`comp` 우선 사용을 원칙으로 한다.

## 6. 타이포/레이아웃 원칙

- 기본 폰트: `Pretendard`, `Noto Sans KR`
- 타입은 숫자형 크기 토큰보다 의미형(`title/body/label`) 우선 사용
- 모바일 우선 단일 컬럼 설계
- 기본 최대 폭: `1200px`
- 간격 스케일: `4/8/12/16/24/32/48`

## 7. 상태 표현 원칙

- 상태는 색상 + 텍스트/아이콘/레이블을 함께 사용한다.
- 상태 우선순위는 `error > warning > success > neutral`로 본다.
- 추천 도메인(`buy-now`, `buy-wait` 등)은 시스템 상태 색상과 연결해 유지한다.

## 8. 접근성 기준

- 텍스트 대비는 WCAG AA 이상을 목표로 한다.
- 포커스 링은 키보드 탐색 시 항상 보이도록 유지한다.
- 최소 인터랙션 타겟은 `40x40` 이상을 지향한다.
- 색상만으로 의미를 전달하지 않는다.

## 9. 변경 거버넌스

- 디자인 언어 변경은 "원칙 문서 업데이트 -> 토큰 사전 업데이트 -> 컴포넌트 계약 업데이트" 순서로 진행한다.
- breaking change는 반드시 마이그레이션 가이드를 동반한다.
- 신규 토큰 추가 시 "목적, 사용 컴포넌트, 대체되는 기존 토큰"을 문서에 기록한다.
