# Migration Playbook

## 1. 목적

- 기존 프론트 화면을 새 디자인 랭귀지로 일괄 전환하기 위한 실행 절차를 정의한다.
- 이 문서는 "한 번에 바꾸되, 안전하게"를 목표로 한다.

## 2. 전제

- 기준 토큰 파일: `src/styles/tokens.css`
- 기준 문서:
- `docs/design-language.md`
- `docs/tokens-spec.md`
- `docs/component-contract.md`

## 3. 전환 원칙

- 화면 단위가 아니라 토큰 계층 단위로 먼저 안정화한다.
- 새 토큰 도입 후 컴포넌트를 일괄 치환한다.
- legacy alias는 전환 완료 전까지 유지한다.

## 4. 단계별 계획

### Phase 0. 동결

- 신규 스타일 하드코딩 금지 정책을 선언한다.
- UI 변경 PR은 토큰 사용 여부를 필수 체크한다.

### Phase 1. 인벤토리

- 기존 스타일 자산을 분류한다.
- raw color
- direct spacing/radius 값
- inline style
- 컴포넌트별 상태 누락

### Phase 2. 매핑표 작성

- 기존 스타일 값을 `sys`/`comp`/`semantic` 토큰으로 매핑한다.
- 매핑 불가 항목은 신규 토큰 후보로 등록한다.

### Phase 3. 토큰 확정

- 신규 토큰을 `tokens.css`에 추가한다.
- 네이밍/계층 규칙 위반 여부를 검토한다.
- deprecated alias 목록을 정의한다.

### Phase 4. 컴포넌트 일괄 치환

- 우선순위:
- 공통 컴포넌트(버튼/입력/카드)
- 페이지 레이아웃
- 도메인 컴포넌트
- 치환 시 `component-contract` 준수 여부를 함께 검증한다.

### Phase 5. 접근성 검증

- 대비, 포커스, 키보드 탐색, 클릭 타겟을 점검한다.
- 의미 전달이 색상에만 의존하는 지점을 제거한다.

### Phase 6. 정리

- 미사용 토큰 제거
- legacy alias 단계적 제거
- 문서와 실제 구현 불일치 항목 정리

## 5. PR 운영 규칙

- PR 템플릿 필수 항목:
- 사용/변경 토큰 목록
- 제거한 하드코딩 스타일 목록
- 접근성 영향
- 시각적 비교 스크린샷

## 6. 완료 기준 (Definition of Done)

- 주요 화면에서 raw color 하드코딩 0건
- 핵심 컴포넌트가 `component-contract`를 충족
- 대비/포커스/상태 표현 기준 충족
- 문서와 토큰 구현이 일치

## 7. 롤백 전략

- 변경은 화면군 단위로 배포한다.
- 심각한 가독성/접근성 이슈 발생 시 legacy alias 경로로 임시 복귀한다.
- 롤백 후 원인 분석 항목을 문서화한다.

## 8. 리스크와 대응

- 리스크: 대규모 일괄 치환 중 누락/파손
- 대응: 컴포넌트 우선 치환 + 체크리스트 기반 검수
- 리스크: 토큰 과증식
- 대응: 신규 토큰 생성 전 기존 토큰 재사용 검토 의무화
